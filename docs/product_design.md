# 产品设计文档

## 1. 项目概述

`go-resumes-record` 是一个面向个人求职过程管理的后端项目，核心能力是记录岗位投递、公司信息、招聘进度、HR 反馈和投递渠道效果，帮助用户把零散的求职动作沉淀为可查询、可复盘、可统计的数据。

项目当前定位为 **个人求职投递记录系统 MVP**，优先保证数据结构清晰、业务流程闭环、接口可扩展，而不是追求复杂的前端展示或过早引入 AI 自动化。

---

## 2. 项目背景

在求职过程中，岗位来源分散、岗位要求差异大、投递节奏不稳定、HR 反馈难以追踪，容易出现以下问题：

- 投递记录散落在招聘软件、微信、表格、邮件中，无法统一管理。
- 不清楚哪些岗位已经投递、哪些需要跟进、哪些已经失效。
- 无法统计不同投递渠道的有效性，例如 BOSS 直聘、猎聘、内推、微信群等。
- 面试反馈、拒绝原因、岗位要求没有沉淀，难以指导简历和项目优化。
- 长期求职过程中，无法判断自己的投递方向是否正确。

因此，本项目通过 Go 后端 + MySQL 数据库实现一个结构化求职记录系统，用于辅助个人求职管理和后续复盘。

## 3. 目标用户

### 3.1 当前用户

- 项目作者本人。
- 正在找 Go 后端、后端开发、运维、测试、网络工程师等岗位的个人求职者。

### 3.2 潜在用户

- 想系统管理岗位投递记录的应届生或转岗求职者。
- 想统计投递转化率、渠道有效性、面试反馈的求职者。
- 后续可扩展给小型求职训练营、就业辅导场景使用。

## 4. 核心问题

| 问题 | 说明 | 产品应对方式 |
| --- | --- | --- |
| 投递信息分散 | 岗位、公司、HR、反馈分散在不同平台 | 用统一表结构记录投递信息 |
| 状态不可追踪 | 不知道哪些岗位等待回复、哪些需要跟进 | 用投递状态和下次跟进时间管理 |
| 渠道效果不清楚 | 不知道哪个渠道带来更多有效反馈 | 统计不同渠道的投递数、回复数、面试数 |
| 面试反馈无法复盘 | 面试问题、拒绝原因没有沉淀 | 用招聘进度表记录每次沟通和反馈 |
| 求职方向难调整 | 不清楚岗位要求和自身能力差距 | 后续基于岗位信息做关键词和能力缺口分析 |

## 5. 产品目标与非目标

### 5.1 产品目标

1. 支持记录公司、岗位、投递渠道、联系人、投递时间、当前状态。
2. 支持记录每次招聘进度，例如简历筛选、一面、二面、HR 面、Offer、拒绝、放弃。
3. 支持查询“今天需要跟进的岗位”。
4. 支持按投递状态、最终状态、投递渠道、公司、时间范围进行筛选。
5. 支持后续统计投递转化率和渠道有效性。
6. 支持后续接入 AI 做岗位信息补全、岗位关键词提取和简历优化建议。

### 5.2 当前阶段非目标

1. 不做完整招聘平台，不提供岗位爬取和职位发布功能。
2. 不做多人协作和权限系统，当前默认单用户使用。
3. 不做复杂 CRM，联系人信息先冗余存储在投递记录和进度记录中。
4. 不在 MVP 阶段接入 AI 自动投递或自动采集。
5. 不优先做复杂前端，当前以后端接口和数据库设计为主。

## 6. 核心业务对象

| 业务对象 | 数据表 | 说明 |
| --- | --- | --- |
| 公司 | `companies` | 记录公司名称、行业、规模、地址、官网等基础信息 |
| 岗位投递记录 | `work_info_records` | 记录岗位、公司、投递渠道、联系人、当前状态、最终结果 |
| 招聘进度节点 | `recruitment_progress_records` | 记录每一次沟通、面试、反馈和下一步动作 |

表关系：

```text
companies 1 ─── N work_info_records 1 ─── N recruitment_progress_records
```

## 7. 核心业务流程

### 7.1 新增岗位投递记录

```text
输入公司信息
  ↓
判断公司是否已存在
  ↓
不存在则创建 companies
  ↓
创建 work_info_records
  ↓
状态默认为：未投递 / 暂无最终结果
```

示例：

```text
公司：广州某某科技有限公司
岗位：Go 后端开发工程师
渠道：BOSS直聘
地点：广州天河
学历要求：本科
投递状态：未投递
```

### 7.2 记录投递动作

```text
选择岗位记录
  ↓
填写投递时间、投递渠道、联系人
  ↓
更新 send_resume_result 为等待回复或投递成功
  ↓
设置 next_follow_up_time
```

示例：

```text
2026-07-16 通过 BOSS 直聘发送简历
HR 承诺 3 天内反馈
下次跟进时间：2026-07-19
```

### 7.3 记录招聘进度

```text
选择岗位记录
  ↓
新增 recruitment_progress_records
  ↓
记录节点状态、沟通内容、下一步动作
  ↓
必要时同步更新 work_info_records 当前状态
```

示例：

```text
节点：一面
时间：2026-07-20 15:00
内容：问了 Go 并发、MySQL 索引、项目部署
下一步：等待二面通知
```

### 7.4 跟进提醒

```text
查询 next_follow_up_time <= 今天 且 final_status 非结束态的记录
  ↓
展示需要跟进的岗位
  ↓
用户完成跟进后新增进度节点
  ↓
更新下次跟进时间或最终状态
```

结束态包括：

```text
Offer、被拒、主动放弃、失联、已入职
```

### 7.5 投递渠道复盘

```text
按时间范围统计渠道数据
  ↓
统计投递数、等待回复数、面试数、Offer 数、被拒数
  ↓
计算回复率、面试率、Offer率
  ↓
判断渠道是否值得继续投入时间
```

## 10. 关键业务规则

### 10.1 公司创建规则

- 新增投递记录时，必须关联公司。
- 如果公司已存在，直接使用已有 `company_id`。
- 如果公司不存在，先创建公司，再创建投递记录。
- 公司唯一性优先以 `company_name` 判断。

### 10.2 投递状态规则

| 状态 | 说明 | 是否结束态 |
| --- | --- | --- |
| 未投递 | 只记录了岗位，还没发送简历 | 否 |
| 等待回复 | 已发送简历，等待 HR 或平台反馈 | 否 |
| 投递成功 | 简历已被接收或进入流程 | 否 |
| 投递失败 | 简历发送失败、岗位关闭、明显不匹配 | 是 |

### 10.3 最终状态规则

| 状态 | 说明 | 是否结束态 |
| --- | --- | --- |
| 暂无 | 尚无最终结果 | 否 |
| 流程中 | 已进入面试或沟通流程 | 否 |
| Offer | 拿到 Offer | 是 |
| 被拒 | 企业明确拒绝 | 是 |
| 主动放弃 | 用户主动终止流程 | 是 |
| 失联 | 长时间无反馈，不再跟进 | 是 |
| 已入职 | 最终入职 | 是 |

### 10.4 跟进规则

- 当 `next_follow_up_time <= 当前日期` 且最终状态不是结束态时，该记录进入待跟进列表。
- 每次跟进都应新增一条 `recruitment_progress_records`，不要只覆盖主表备注。
- 如果 HR 给出明确下一次反馈时间，需要同步更新主表 `next_follow_up_time`。

### 10.5 渠道可靠性判断规则

渠道统计以时间窗口为单位，例如近 7 天、近 30 天。

建议指标：

```text
回复率 = 有回复数量 / 投递数量
面试率 = 面试邀请数量 / 投递数量
Offer率 = Offer数量 / 投递数量
```

初步规则：

- 投递数量过少时不做判断，例如少于 10 条只展示数据，不打标签。
- 连续 30 天回复率明显低于其他渠道，可标记为“低效渠道”。
- 面试率高的渠道优先投入时间。

## 11. 数据统计指标

| 指标 | 说明 | 数据来源 |
| --- | --- | --- |
| 总投递数 | 时间范围内投递记录数量 | `work_info_records` |
| 等待回复数 | `send_resume_result = 等待回复` 的数量 | `work_info_records` |
| 流程中数量 | `final_status = 流程中` 的数量 | `work_info_records` |
| Offer 数 | `final_status = Offer` 的数量 | `work_info_records` |
| 被拒数 | `final_status = 被拒` 的数量 | `work_info_records` |
| 待跟进数 | 到达 `next_follow_up_time` 且未结束的数量 | `work_info_records` |
| 渠道回复率 | 按渠道统计有反馈记录的比例 | `work_info_records` + `recruitment_progress_records` |
| 渠道面试率 | 按渠道统计进入面试节点的比例 | `recruitment_progress_records` |

## 12. 输入校验与异常处理

| 场景 | 校验规则 | 错误提示建议 |
| --- | --- | --- |
| 创建岗位记录 | 岗位名称不能为空 | `job_name is required` |
| 创建岗位记录 | 公司名称或公司 ID 必须存在一个 | `company is required` |
| 枚举字段 | 只能传定义范围内的值 | `invalid enum value` |
| 投递时间 | 不能明显晚于当前时间太多 | `invalid send_resume_time` |
| 跟进时间 | 不能早于记录创建时间太多 | `invalid next_follow_up_time` |
| 查询详情 | 记录不存在 | `work record not found` |
| 新增进度 | 投递记录不存在 | `work record not found` |

## 13. 隐私与安全要求

1. 联系方式、HR 微信、手机号、邮箱属于敏感信息，不应写入公开 README、日志或错误信息。
2. 接口日志只记录请求 ID、接口路径、耗时和状态码，不直接打印完整请求体。
3. 后续如果部署到公网，需要增加认证机制，例如 JWT 或 Basic Auth。
4. 数据库连接密码只能放在 `.env` 或部署环境变量中，不提交到 Git。
5. 后续接入 AI 时，默认不要把联系人、手机号、邮箱发送给外部模型。

## 14. 版本规划

### V1：基础记录闭环

目标：完成从公司、岗位、投递、跟进到结果复盘的基础闭环。

- 公司表和岗位投递表落地。
- 支持新增、查询、更新岗位投递记录。
- 支持新增招聘进度节点。
- 支持待跟进列表。
- 支持基础状态统计。

### V2：统计分析增强

目标：让系统能辅助判断投递策略是否有效。

- 渠道转化率统计。
- 公司维度统计。
- 岗位类型统计。
- 关键词统计。
- 简历版本与投递结果关联。

### V3：AI 辅助

目标：减少人工录入，提高求职复盘质量。

- AI 提取岗位关键词。
- AI 总结岗位能力要求。
- AI 根据失败反馈生成简历修改建议。
- 半自动补全公司信息。
- 半自动生成求职周报。

> 基于 Go + Gin + GORM + MySQL 实现个人求职投递管理系统，支持公司信息管理、岗位投递记录、招聘进度时间线、待跟进提醒和渠道转化统计。通过结构化数据沉淀求职过程，辅助分析投递策略和简历优化方向。
